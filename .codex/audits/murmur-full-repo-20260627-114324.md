## Executive Summary

Murmur builds and its current Vitest suite passes, but the audit found several real defects in high-risk areas:

- Dictation lifecycle cancellation is not a hard cancellation. A recording cancelled during STT/LLM/paste can still paste text and write history.
- Very fast stop after recording start can leave the main process in `transcribing` with no renderer completion path.
- Packaged apps honor `ELECTRON_RENDERER_URL`, and every window exposes the full privileged preload bridge. If a packaged launch environment is contaminated, a remote or alternate renderer can drive provider settings, history, dictation, local-data deletion, and paste actions.
- State snapshots and privileged preload APIs are over-broad for pill and mode-selector windows, carrying API keys and history through surfaces that do not need them.
- Provider privacy classification trusts renderer-controlled `isCloud` / `isLocal` booleans, so custom remote endpoints can be treated as local.
- The advertised `textRetentionDays` privacy setting is not enforced.
- Release packaging depends on ignored local artifacts, and every current STT runtime catalog URL returned HTTP 404 during the audit.

The strongest remediation theme is to reduce trust in renderer-originated state and make long-running recording/download/storage operations cancellable, bounded, and role-scoped.

## Findings

### DA-001 High: Cancelled recordings can still paste and write history

**Evidence**

- `cancelRecording` replaces the session with a new cancelled id and clears context, but it does not abort in-flight STT, LLM, or paste work: `src/main/app-controller.ts:875`.
- `completeRecording` checks the session id before context capture and once after context capture, then performs STT, optional LLM processing, paste, and history writes without another cancellation/session-token check: `src/main/app-controller.ts:950`, `src/main/app-controller.ts:971`, `src/main/app-controller.ts:988`, `src/main/app-controller.ts:1015`, `src/main/app-controller.ts:1047`.
- Streaming deltas mutate `this.session` inside the STT callback without rechecking that the session is still current: `src/main/app-controller.ts:977`.

**Impact**

A user who cancels after the renderer has submitted audio can still have dictated text inserted into the focused application and persisted into history. This is a direct correctness and privacy failure for a system-wide dictation tool.

**Recommended fix**

Introduce a per-recording cancellation token or operation object. Pass an `AbortSignal` into STT/LLM providers where possible, recheck the session id before each state transition and before paste/history writes, and make cancellation settle the operation as terminal.

### DA-002 High: Quick stop before microphone setup can leave the app stuck in `transcribing`

**Evidence**

- Main sends `recording:start` and immediately considers the session active: `src/main/app-controller.ts:854`.
- `stopRecording` switches the session to `transcribing` immediately after sending `recording:stop`: `src/main/app-controller.ts:864`.
- In the renderer, `sessionRef.current` is set before `getUserMedia`, but `recorderRef.current` is assigned only after microphone access and recorder setup finish: `src/renderer/src/hooks/useRecordingBridge.ts:56`.
- `completeCurrentRecording` returns early when `recorderRef.current` is null, while the `finally` path only stops tracks and clears the local completion flag: `src/renderer/src/hooks/useRecordingBridge.ts:30`.

**Impact**

If a user toggles stop or releases push-to-talk before `getUserMedia` and `startWavRecorder` complete, the renderer has no audio payload to send and the main process remains in a terminal-looking `transcribing` state. The session can become stuck until another error path or restart.

**Recommended fix**

Track a main-process `starting` or `awaiting-recorder` phase, require renderer acknowledgement before stop transitions to `transcribing`, or make the renderer report an explicit recording error/cancel when stop arrives before a recorder exists.

### DA-003 High: Packaged builds can load an arbitrary renderer from `ELECTRON_RENDERER_URL`

**Evidence**

- The renderer loader uses `process.env.ELECTRON_RENDERER_URL` for every window without gating it to development mode: `src/main/app-controller.ts:390`.
- That renderer receives the full preload API through `contextBridge.exposeInMainWorld("murmur", api)`: `src/preload/index.ts:24`, `src/preload/index.ts:149`.
- The exposed API includes provider mutation, model download/deletion, dictation start/stop/cancel, complete recording, history copy/repaste/delete/clear, and local data clear: `src/preload/index.ts:28`, `src/preload/index.ts:33`, `src/preload/index.ts:43`, `src/preload/index.ts:56`, `src/preload/index.ts:67`, `src/preload/index.ts:73`.

**Impact**

If a packaged app is launched with `ELECTRON_RENDERER_URL` set by the environment, it can load a non-packaged renderer with the app's privileged preload bridge. This widens an environment-variable mistake into full local app control.

**Recommended fix**

Only honor `ELECTRON_RENDERER_URL` in explicit dev mode, for example under `!app.isPackaged` or a signed internal development switch. In packaged mode, always load the bundled `index.html`.

### DA-004 High: All windows share an over-broad privileged preload and state surface

**Evidence**

- The app exposes a single full preload API, not role-specific APIs: `src/preload/index.ts:24`.
- The main, pill, and mode-selector windows are loaded through the same renderer/preload path: `src/main/app-controller.ts:181`, `src/main/app-controller.ts:232`, `src/main/app-controller.ts:270`.
- `AppStateSnapshot` includes provider configs with `apiKey`, full history, model library, setup, session, and capabilities: `src/shared/types.ts:171`, `src/shared/types.ts:187`, `src/shared/types.ts:305`.
- `broadcastState` sends the full app state to the main window and derived pill/mode-selector updates to auxiliary windows; `getModeSelectorSnapshot` builds from `getSnapshot` when no snapshot is passed: `src/main/app-controller.ts:1371`, `src/main/app-controller.ts:1388`, `src/main/app-controller.ts:1443`.

**Impact**

Small auxiliary renderer surfaces run with the same callable authority as the settings/history UI. They do not need API keys, history mutation, local-data deletion, provider updates, or raw recording completion. Any renderer compromise, XSS, dependency issue, or accidental route exposure in those windows has unnecessary blast radius.

**Recommended fix**

Split preload APIs and IPC channels by window role. Return redacted snapshots by default, especially removing API keys from all renderer state. Make pill and selector windows receive only session/theme/mode-selector fields and only the commands they need.

### DA-005 High: Cloud/local privacy classification trusts renderer-controlled provider booleans

**Evidence**

- Provider configs store `isCloud` and `isLocal` as mutable config fields: `src/shared/types.ts:171`, `src/shared/types.ts:187`.
- Storage normalization preserves renderer-provided `isCloud` / `isLocal` booleans: `src/main/services/storage.ts:725`, `src/main/services/storage.ts:763`.
- Credential gating trusts `!provider.isCloud` as sufficient: `src/shared/model-activation.ts:36`.
- Session and history disclosure flags copy those booleans directly: `src/main/app-controller.ts:849`, `src/main/app-controller.ts:1030`, `src/main/app-controller.ts:1035`.
- STT/LLM requests send audio or prompt data to configured base URLs: `src/main/services/stt.ts:216`, `src/main/services/llm.ts:61`.

**Impact**

A custom remote endpoint can be stored as local by setting `isCloud: false`, causing the UI/session/history to under-report cloud use and bypass cloud credential assumptions. That contradicts the privacy model documented for opt-in cloud providers.

**Recommended fix**

Derive cloud/local classification in main from provider type and URL, not from renderer input. Treat non-loopback HTTP(S) endpoints as remote/cloud unless explicitly modeled and confirmed. Keep user-visible disclosure based on derived classification.

### DA-006 Medium: Critical IPC inputs are not consistently validated or bounded in main

**Evidence**

- `dictation:complete-recording` accepts a typed payload directly from the renderer and calls `completeRecording`; the schema exists but is not used here: `src/main/app-controller.ts:522`, `src/shared/schemas.ts:398`.
- `completeRecording` constructs `new Uint8Array(payload.audio)` before entering its `try` block and without checking byte size or MIME allowlist: `src/main/app-controller.ts:950`, `src/main/app-controller.ts:964`.
- State-mutating handlers for settings, modes, and providers accept renderer objects directly, relying on TypeScript types rather than runtime validation at the IPC boundary: `src/main/app-controller.ts:417`, `src/main/app-controller.ts:433`, `src/main/app-controller.ts:444`.

**Impact**

A compromised renderer or buggy route can submit oversized audio buffers, malformed settings/provider payloads, or invalid mode data. The result ranges from main-process memory pressure to persisted invalid configuration.

**Recommended fix**

Validate every IPC handler in main with shared schemas. Add explicit size caps for audio payloads, accepted MIME types, and maximum counts/lengths for modes, rules, providers, vocabulary, and history operations.

### DA-007 Medium: `textRetentionDays` is exposed but not enforced

**Evidence**

- The default setting is 90 days: `src/shared/defaults.ts:13`.
- The configuration UI exposes `Text retention days`: `src/renderer/src/views/ConfigurationView.tsx:264`.
- Storage only caps history by count with `.slice(0, 2000)`: `src/main/services/storage.ts:241`.
- Search found no enforcement outside type/schema/UI references:
  - `src/shared/types.ts:225`
  - `src/shared/defaults.ts:15`
  - `src/shared/schemas.ts:198`
  - `src/main/services/storage.ts:80`
  - `src/renderer/src/views/ConfigurationView.tsx:264`
- The docs acknowledge that current storage reads and writes the latest 2000 items: `docs/architecture/security-and-privacy.md:22`.

**Impact**

Users can set a privacy retention period that has no effect. Transcript text and context metadata can persist far beyond the configured window.

**Recommended fix**

Apply retention during history reads and writes, and when `textRetentionDays` changes. Define `0` semantics explicitly, then test age-based pruning for SQLite and JSON backends.

### DA-008 Medium: Downloaded STT model files are not integrity-checked

**Evidence**

- The offline model catalog defines remote direct-file and archive URLs without hashes or signatures: `src/shared/model-catalog.ts:147`, `src/shared/model-catalog.ts:195`.
- `downloadDirectFile` fetches, writes, and renames the file without checksum verification: `src/main/services/model-library.ts:164`.
- `downloadArchive` fetches and extracts archives without checksum verification: `src/main/services/model-library.ts:243`.
- Archive extraction shells out to `tar -xjf` directly into the model directory: `src/main/services/model-library.ts:878`.

**Impact**

Local STT models come from third-party hosts but are trusted after TLS alone. A compromised upstream release asset, CDN edge, or redirected download can alter local model behavior or place unexpected files during extraction.

**Recommended fix**

Add SHA-256 or signature metadata for every downloadable model artifact, verify before activation/extraction, and extract archives into a temporary directory with path traversal checks before moving the expected model directory into place.

### DA-010 Medium: Release packaging depends on ignored local artifacts

**Evidence**

- `pack` and `dist` stage runtimes from local vendor directories before electron-builder: `package.json:22`.
- Packaged resources include `resources/bin/linux-fast-paste` and `.cache/bundled-runtimes/runtimes`: `package.json:40`.
- Both `vendor/runtimes/` and `resources/bin/linux-fast-paste` are ignored: `.gitignore:67`, `.gitignore:69`.
- The staging script only checks that at least one executable candidate exists, then copies the entire source runtime tree: `scripts/stage-bundled-runtimes.mjs:23`, `scripts/stage-bundled-runtimes.mjs:31`.

**Impact**

Two clean builders can produce different packages depending on untracked local runtime/helper state. The current machine has these files, but the repo alone is not sufficient to reproduce the distributable.

**Recommended fix**

Make helper and runtime artifacts reproducible in the packaging pipeline. Build or fetch the helper as part of `pack`/`dist`, verify runtime hashes before staging, and keep ignored local caches separate from release inputs.

### DA-011 Medium: Settings changes rewrite the full history store, and mode selector arrows amplify it

**Evidence**

- `writeState` writes config and history for every state mutation: `src/main/services/storage.ts:368`.
- SQLite history writes delete all rows and FTS rows, then reinsert every item, without an explicit transaction: `src/main/services/storage.ts:378`.
- Mode-selector navigation persists `activeModeId` on every movement: `src/main/app-controller.ts:1255`, `src/main/app-controller.ts:1273`.
- `broadcastState` builds full snapshots as part of these updates: `src/main/app-controller.ts:1443`.

**Impact**

Changing a setting, pressing arrow keys in the mode selector, or toggling preferences can rewrite up to 2000 history entries. This is unnecessary I/O, increases chances of data loss on interruption, and can create UI lag when history grows.

**Recommended fix**

Separate config writes from history writes. For SQLite, update history incrementally inside transactions. For selector navigation, keep a transient highlighted id and persist only on explicit selection if that matches the intended UX.

### DA-012 Medium: Long recordings are buffered and encoded synchronously on the renderer thread

**Evidence**

- Audio processor chunks are appended to an in-memory `Float32Array[]`: `src/renderer/src/hooks/useRecordingBridge.ts:103`, `src/renderer/src/hooks/useRecordingBridge.ts:116`.
- Stop merges all chunks into one `Float32Array`, then synchronously encodes WAV: `src/renderer/src/hooks/useRecordingBridge.ts:147`, `src/renderer/src/hooks/useRecordingBridge.ts:169`, `src/renderer/src/hooks/useRecordingBridge.ts:180`.
- Recording level events are published as often as every 32 ms: `src/renderer/src/hooks/useRecordingBridge.ts:11`, `src/renderer/src/hooks/useRecordingBridge.ts:126`.

**Impact**

Long dictations can allocate large buffers and stall the renderer at stop time. This is most visible right when the user expects fast feedback.

**Recommended fix**

Use streaming/chunked recording through `MediaRecorder` or an `AudioWorklet`/worker pipeline, cap maximum recording length, and perform WAV encoding off the main renderer thread.

### DA-013 Medium: Model download progress persists and emits on every chunk

**Evidence**

- Direct downloads persist and emit progress after each stream chunk: `src/main/services/model-library.ts:194`, `src/main/services/model-library.ts:201`.
- Archive downloads do the same: `src/main/services/model-library.ts:322`, `src/main/services/model-library.ts:329`.
- `persistAndEmit` always writes storage before emitting: `src/main/services/model-library.ts:635`.

**Impact**

Large model downloads can cause hundreds or thousands of config/history writes and state broadcasts. This can degrade UI responsiveness and wear storage unnecessarily.

**Recommended fix**

Throttle progress persistence and broadcasts by time or byte delta. Keep in-memory progress for high-frequency UI updates and persist only coarse checkpoints plus final/error/cancel states.

## Suggested Remediation Order

1. Fix dictation cancellation/session-token checks and the quick-stop startup race.
2. Gate `ELECTRON_RENDERER_URL` to dev mode and split/redact preload/state surfaces.
3. Derive cloud/local classification in main and enforce `textRetentionDays`.
4. Repair STT runtime asset publication and make packaging inputs reproducible.
5. Add model artifact integrity, storage write separation, and progress throttling.
6. Add CI that runs `mise run lint`, `mise run test`, `mise run build`, runtime manifest checks, package smoke checks, and dependency audit with documented exceptions.
